## 2 · Context & Background

Piubelle is a Portuguese home textile company (est. 1981, ~150 employees, Maia/Porto) with ~95% export revenue and ~80% bedding concentration. A 4-person procurement team manages ~200 active suppliers across raw materials and outsourced production stages (weaving, cutting, sewing, finishing) using a fragmented stack: Sage ERP, Outlook, spreadsheets, WhatsApp, and phone. Delays are managed reactively, with no unified visibility or proactive alerting. Known consequences include a 10% OTIF penalty from a missed US vessel and unrecoverable air freight costs from a delayed lace shipment. WARP is a Streamlit prototype that simulates integration with these systems using synthetic data and the Claude API for agent capabilities. It is submitted as a working demo for the Generative AI & AI Agents group project.

## 5 · Architecture Decisions

| Decision | Reason | Rejected alternatives |
|---|---|---|
| Synthetic CSVs over live Sage/Outlook integration | No access to Piubelle production systems; GDPR compliance; demo portability | Mock API wrapper, live ERP connector |
| `random.seed(42)` on all generators | Reproducible data across all team members — same Red/Amber alerts every run | Random seed → inconsistent demo |
| Streamlit multi-page over single-page | Each capability (task board, follow-up, ERP update) is a discrete workflow with its own HITL gate | Single-page with tabs (harder to navigate in pitch) |
| Claude API for drafting only — not autonomous sending | HITL requirement: agent never sends without buyer "Send" confirmation | Full automation (violates GDPR + EU AI Act transparency) |
| pandas over polars | Dataset is small (≤400 POs, 200 suppliers) — no performance case for polars | polars (overkill), SQL (no DB needed) |
| Score dimensions weighted (35 / 20 / 15 / 15 / 10 / 5) | On-time rate is the primary procurement KPI; penalty history captures financial impact | Equal weighting (loses procurement domain logic) |
| Streamlit Community Cloud for deployment | Free tier, direct GitHub integration, no DevOps setup needed | Azure App Service, Heroku, local-only |
| One action at a time (no bulk send) | Prevents compounding errors; matches HITL requirement per action | Batch approve (fails Common Pitfall 3) |

## 6 · Coding Standards (Project-Specific)

> These extend `~/.claude/CLAUDE.md` for this project.

### Naming
- DataFrames: `snake_case` descriptive nouns — `purchase_orders_df`, `scores_df`, not `df`, `df2`
- Streamlit page files: `[N]_[FeatureName].py` — number prefix controls sidebar order
- Agent functions: verb-first — `detect_alerts()`, `compute_scores()`, `draft_followup()`, `generate_report()`
- Session state keys: `snake_case` strings — `st.session_state["erp_preview"]`, `st.session_state["buyer"]`
- Synthetic output files: `snake_case` nouns matching their real-world source — `purchase_orders.csv` (not `pos.csv`)

### Data Rules
- `data/synthetic/` files are **read-only** at runtime — never written to by app pages
- All CSV reads use `st.cache_data` in `utils/data_loader.py` — no raw `pd.read_csv()` in page files
- Date columns must be cast to `.dt.date` before any comparison with `date.today()` — never compare mixed types
- All monetary values in **EUR**
- Delay days must be `int`, never `float` — use `int((expected - today).days)`

### Agent / HITL Rules
- Every Claude API call is wrapped in a `try/except` with a user-visible fallback message — never let an API error crash the page
- HITL gates are enforced via `st.session_state` flags, never skipped by default values
- `claude_agent.py` functions accept structured dicts (not formatted strings) as input — easier to test and update prompts
- Draft outputs are displayed in `st.text_area` (editable) before any approval button is shown — buyer can always modify
- Approval buttons use `type="primary"` — rejection/discard uses default style — visual hierarchy enforces the gate

### Scoring Rules
- Normalize each dimension to [0, 1] before weighting — never weight raw values
- Final score = `(weighted_sum * 9 + 1).round(1)` — always maps to [1.0, 10.0]
- Trend arrow: `↑` if last-90-day on-time rate improved > 5pp, `↓` if deteriorated > 5pp, `→` otherwise
- Suppliers with fewer than 5 historical events display score as `N/A` — insufficient data

### Streamlit Rules
- No raw `st.write()` for structured data — use `st.metric`, `st.dataframe`, or `st.plotly_chart`
- `st.session_state` must be initialized in `Home.py` before any page reads it — guard with `if key not in st.session_state`
- No `st.experimental_rerun()` — use `st.switch_page()` for page transitions
- All Plotly charts use `use_container_width=True`

## 11 · Known Issues & Constraints

- [ ] WhatsApp is not integrated — buyers must relay WhatsApp updates as manual input; this is a documented WARP limitation, not a bug
- [ ] Sage ERP write access is simulated — the "Confirm ERP update" gate updates session state only; in production this would call the Sage API
- [ ] Outlook send is simulated — approved follow-up drafts are logged in session state only; no SMTP integration in prototype
- [ ] Supplier scores for newly onboarded suppliers (< 5 delivery events in history) are marked `N/A` — score reliability caveat documented in the UI
- [ ] Claude API latency: draft generation typically 3–8s; if demo environment is slow, use pre-generated mock outputs as fallback (see `agent/claude_agent.py` mock mode)
- [ ] Streamlit session state resets on browser refresh — warn demo audience; this is a prototype constraint, not a production architecture
- [ ] `data/synthetic/` files use `random.seed(42)` — all team members must use the same seed or alert counts will differ across machines

## 12 · Out of Scope

> Claude should NOT suggest or implement the following for this project:

- Live Sage ERP API integration — prototype uses synthetic CSVs only
- Live Outlook / Microsoft Graph API integration — email threads are synthetic
- WhatsApp Business API — out of scope and GDPR-sensitive
- Docker / containerisation — Streamlit Community Cloud handles deployment
- Real-time / streaming data — session-driven batch scans only
- User authentication / login system — buyer identity is set via a `st.selectbox` at session start
- Database (SQL/NoSQL) — all state is in-session and in flat files
- Automated email sending — HITL gate means all sends are buyer-confirmed, manual actions
- Any processing of real Piubelle supplier data, contracts, or financial records — GDPR constraint
